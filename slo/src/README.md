# SLO Workload CLI

A load-generation tool that drives read/write traffic against a YDB table and pushes metrics
to Prometheus. Used to measure SDK reliability under cluster-level fault injection.

## Commands

### `create` — Create the test table

```bash
slo create <endpoint> <db> [options]
```

| Argument / Option | Default | Description |
|---|---|---|
| `<endpoint>` | — | YDB endpoint, e.g. `grpc://localhost:2136` |
| `<db>` | — | YDB database path, e.g. `/local` |
| `-t, --table-name` | `testingTable` | Table name |
| `--min-partitions-count` | `6` | Minimum partition count |
| `--max-partitions-count` | `1000` | Maximum partition count |
| `--partition-size` | `1` | Partition size in MB |
| `-c, --initial-data-count` | `1000` | Number of rows to pre-populate |
| `--write-timeout` | `10000` | Write timeout (ms) |

**Example:**
```bash
slo create grpcs://ydb.example.com:2135 /some/folder \
  -t testingTable --min-partitions-count 6 -c 1000
```

### `cleanup` — Drop the test table

```bash
slo cleanup <endpoint> <db> [options]
```

| Argument / Option | Default | Description |
|---|---|---|
| `<endpoint>` | — | YDB endpoint |
| `<db>` | — | YDB database path |
| `-t, --table-name` | `testingTable` | Table name to drop |
| `--write-timeout` | `10000` | Write timeout (ms) |

### `run` — Execute the workload

```bash
slo run <endpoint> <db> [options]
```

| Argument / Option | Default | Description |
|---|---|---|
| `<endpoint>` | — | YDB endpoint |
| `<db>` | — | YDB database path |
| `-t, --table-name` | `testingTable` | Table name |
| `--prom-pgw` | **(required)** | Prometheus Pushgateway URL, e.g. `http://localhost:9091` |
| `--report-period` | `250` | Metrics push interval (ms) |
| `--read-rps` | `1000` | Target read requests per second |
| `--read-timeout` | `10000` | Read timeout (ms) |
| `--write-rps` | `100` | Target write requests per second |
| `--write-timeout` | `10000` | Write timeout (ms) |
| `--time` | `140` | Total run duration (seconds) |
| `--shutdown-time` | `30` | Grace period before force-killing workers (seconds) |
| `-c, --initial-data-count` | `1000` | Row count (used by the read job for key range) |

**Example:**
```bash
slo run grpcs://ydb.example.com:2135 /some/folder \
  --prom-pgw http://localhost:9091 \
  --read-rps 1000 --write-rps 100 --time 600 --shutdown-time 30
```

## Table Schema

```sql
CREATE TABLE testingTable (
    id       Uint64,
    hash     Uint64,  -- Digest::NumericHash(id)
    payload_str       Utf8,
    payload_double    Double,
    payload_timestamp Timestamp,
    payload_hash      Uint64,
    PRIMARY KEY (hash, id)
);
```

## What the Workload Does

Two concurrent jobs run simultaneously:

- **`readJob`** — reads rows by random IDs (keys generated by `writeJob`)
- **`writeJob`** — generates and upserts new rows

## Collected Metrics (pushed to Prometheus)

| Metric | Description |
|---|---|
| `oks` | Number of successful requests |
| `not_oks` | Number of failed requests |
| `inflight` | Requests currently in flight |
| `latency` | Request latency summary (ms) |
| `attempts` | Number of retry attempts per request |

> Reset metrics in Prometheus before and after a run to avoid stale data skewing dashboards.

## Authentication

The workload uses **anonymous credentials** by default (suitable for a local YDB instance).
